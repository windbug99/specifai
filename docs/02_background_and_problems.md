# Specifai - 2. 배경 및 문제 정의

## 2.0 시장 환경 변화: 소규모화 되는 프로젝트 시장

### 핵심 트렌드

IT 서비스 구축 시장은 급속히 소규모화되고 있습니다:

#### 1. 기업 조직구조의 축소
```
과거: 대규모 개발팀 (수십~수백 명)
현재: 소규모 팀 (3-5명)
미래: 1인 또는 프리랜서 조합
```

#### 2. AI의 등장으로 인한 생산성 혁신
```
1명의 개발자가 과거 5명의 업무를 소화
→ 개인 프로젝트 추진 가능
→ 프리랜서 시장 급성장
```

#### 3. 소규모 프로젝트의 증가
```
대규모 프로젝트 감소
소규모 프로젝트 급증
실제로 여러 소규모 프로젝트 = 1개 대규모 프로젝트
```

### Specifai의 포지셔닝

**Specifai는 소규모 프로젝트 시장 중심으로 설계되었습니다:**

- 🎯 **1차 타겟**: 1인 개발자, 프리랜서 (소규모 프로젝트 시장)
- 📈 **향후 확장**: 규모있는 기업 내 팀 (소규모 프로젝트들이 모여 대규모 프로젝트 형성)

이러한 구조에서 **소규모 프로젝트 시장에서 표준이 되는 것이 결국 대규모 프로젝트의 표준**이 됩니다.

## 2.1 요구사항 명세서의 역할

### 전통적 역할 (교과서적 정의)
1. **커뮤니케이션 도구**: 이해관계자 간 합의점
2. **계약 문서**: "이것만 만들면 됨"의 명확한 정의
3. **개발 가이드**: 개발자가 따를 청사진
4. **테스트 기준**: "이게 맞는지" 검증 도구
5. **프로젝트 범위 정의**: 스코프 크리프 방지

### 실제 역할 (현실)
- **사고 구조화 도구** (80%): 애매한 생각을 명확하게 만드는 과정
- **위험 발견 도구** (15%): 구현 전 문제점 발견
- **커뮤니케이션** (5%): 실제 합의 도구로서의 역할

### 이상과 현실의 격차
```
이상: 명세서 작성 → 개발 → 검증 → 완료
현실: 애매한 요구 → 개발 시작 → 분쟁 → 재작업 → 추가 비용
```

## 2.2 현재 시장의 문제점

### 대기업/조직의 문제

#### 현황
- 명세서 작성에 2-3개월 소요
- 작성 후에도 요구사항 지속 변경
- 결과: 명세서와 실제 제품 불일치
- **명세서의 가치 하락**

#### 원인 분석
1. **과도한 절차**: 너무 많은 이해관계자의 참여
2. **변경 관리 실패**: 지속적인 요구사항 변경을 반영하지 못함
3. **형식적 문서화**: 실제 개발과 동떨어진 문서 작성

### 외주/프리랜서 시장의 문제

외주 시장에서는 **요구사항 발주자**와 **실무자** 간의 커뮤니케이션 문제가 핵심입니다.

#### 전형적인 분쟁 시나리오
```
❌ 현재 상황:

요구사항 발주자: "쇼핑몰 만들어주세요"
실무자: "네, 500만원입니다"

3주 후:
요구사항 발주자: "쿠폰 기능도 넣어주세요"
실무자: "그건 추가 비용입니다"
요구사항 발주자: "쇼핑몰이면 당연히 있는 거 아닌가요?"
실무자: "명세서에 없었잖아요"
요구사항 발주자: "명세서에 쇼핑몰이라고 했잖아요"

→ 분쟁 → 미수금 → 나쁜 리뷰 → 신뢰 파괴
```

#### 근본 원인

**발주자 측**:
- 개발에 대한 이해 부족으로 명확한 요구사항 전달 어려움
- 서비스 구축에 대한 대부분을 실무자에게 떠넘김
- 원하는 결과물을 얻기 어려움

**실무자 측**:
- 발주자의 불명확한 요구사항으로 인한 혼란
- 명확한 요구사항 이해를 위해 수많은 회의 필요
- 불명확한 요구사항으로 개발된 불만족스러운 결과물

**공통 문제**:
머리속에 있는 추상적인 아이디어 또는 결과물을 실제 코드로 구현하기 위해서는 구체화가 필요합니다. 보통은 이 단계를 수많은 회의를 통해 해결하려고 하지만, 각각 다른 관점의 사람간의 대화는 수많은 오해와 몰이해로 인해 사고를 정렬하기 어렵습니다.

#### 분쟁 발생 원인 분석

| 원인 | 비율 | 예시 |
|------|------|------|
| **범위 불명확** | 45% | "로그인 기능"이 소셜 로그인 포함인지 불명확 |
| **용어 해석 차이** | 25% | "빠른 속도"의 기준이 다름 |
| **암묵적 기대** | 20% | "당연히 있을 거라 생각했어요" |
| **변경 관리 부재** | 10% | 추가 요청의 비용/일정 미합의 |

### 기존 해결책의 한계

#### 1. ChatGPT 활용
- **시도**: "쇼핑몰 명세서 만들어줘"
- **결과**: 너무 일반적이고 애매한 내용
- **문제점**:
  - 외주 계약 맥락 부재
  - 법적 효력 없음
  - 구체적인 범위 정의 부족
  - 비용/일정 연결 없음

#### 2. 템플릿 다운로드
- **시도**: 인터넷에서 템플릿 검색
- **결과**: 빈 칸을 채우기 어려움
- **문제점**:
  - 비개발자가 이해하기 어려운 용어
  - 프로젝트마다 다른 요구사항
  - 가이드 없는 빈 템플릿

#### 3. 외주사에게 맡김
- **시도**: "명세서는 개발사가 작성"
- **결과**: 개발사 편향적 명세서
- **문제점**:
  - 개발하기 쉬운 방향으로 작성
  - 발주자 요구사항 누락
  - 추후 "명세서에 없다" 논리로 활용

#### 4. 구두 합의
- **시도**: "말로 하면 되지"
- **결과**: 기억 차이로 분쟁
- **문제점**:
  - 증거 부재
  - 세부사항 누락
  - 법적 보호 불가능

## 2.3 AI 시대의 변화: AI-Native 명세서의 중요성

### 왜 지금 명세서가 더 중요해지는가?

#### 패러다임 전환
```
과거 (대규모 조직)
명세서 작성 → 어차피 바뀜 → 쓸모없음 → 폐기

현재/미래 (1인/소규모 + AI)
AI-Native 명세서 → AI가 정확히 구현 → 가치 폭발! → 핵심 자산
```

### 핵심 차이점

#### 1. 규모가 작음
- **과거**: 100명 팀, 명세서 유지보수 불가능
- **현재**: 1-5명 팀, 명세서 관리 가능
- **영향**: 명세서가 실제로 유용한 도구가 됨

#### 2. AI 구현 (핵심 변화)
- **과거**: 인간 개발자가 해석하며 구현 (모호함 허용)
- **현재**: AI가 명세서를 직접 실행 (정확성 필수)
- **영향**: **명세서 품질 = 코드 품질**
- **Specifai 차별화**: AI가 이해할 수 있는 instruction prompt 수준의 명세서 작성

#### 3. 외주/계약 관계
- **과거**: 내부 팀, 지속적 소통 가능
- **현재**: 외주/프리랜서, 신뢰 부족
- **영향**: 명세서 = 유일한 계약 근거

### AI-Native 명세서의 특성

**일반 명세서 vs AI-Native 명세서**

| 항목 | 일반 명세서 | AI-Native 명세서 |
|------|-----------|-----------------|
| **구조** | 자유형식 | 체계적, 위계적 |
| **명확성** | 모호함 허용 | 100% 명확 |
| **논리성** | 선택적 | 필수적 |
| **구체성** | 대략적 | 극도로 구체적 |
| **AI 활용** | 재해석 필요 (재시도 5회+) | 직접 실행 (재시도 1회) |
| **미래** | 계약 문서만 가능 | 직접 코드 생성 + AI 에이전트 요청 |

### AI 코딩 도구와 명세서: 명확성이 생산성을 결정

#### 명세서 품질에 따른 AI 성능 차이

| 명세서 품질 | 특징 | AI 재시도 | 개발 시간 | 코드 품질 | 재작업 비용 |
|------------|------|----------|----------|----------|-----------|
| **불명확** | 모호한 표현 | 5-10회 | 2주 | 낮음 | 높음 |
| **보통** | 부분적 명확 | 3-5회 | 1주 | 보통 | 중간 |
| **명확** | 대부분 명확 | 1-2회 | 2-3일 | 높음 | 낮음 |
| **AI-Native** | 완벽하게 명확 (Specifai) | **1회** | **1-2일** | **매우 높음** | **최소** |

#### 실제 예시: 불명확 vs AI-Native 명세

**불명확한 명세**:
```
"빠른 로그인"
→ AI: "3초? 1초? 0.5초?"
→ 개발자가 재시도, 수정, 다시 요청 (5회+)
```

**AI-Native 명세**:
```
"로그인 시간: 페이지 클릭 → 인증 완료까지 2초 이내
- 네트워크 지연 제외한 순처리 시간 기준
- 측정: 성능 테스트 도구로 검증
- 실패 조건: 2초 이상 시 재작업"
→ AI: 명확하게 한 번에 구현
```

### 향후 AI 에이전트 시대

```
현재 (Specifai):
명세서 (AI가 실행 가능한 수준)
  ↓
AI 코딩 도구 (Cursor, Claude)
  ↓
코드 생성

미래 (AI 에이전트 시대):
명세서 (AI-Native instruction prompt)
  ↓
AI 에이전트 (자율 개발)
  ↓
완성된 서비스 (직접 배포)

→ 명세서 품질이 모든 것을 결정
```

#### 실제 예시

##### Case 1: 불명확한 명세
```
프롬프트: "이커머스 사이트 만들어줘"
결과: 기본적인 상품 목록만 있는 사이트
문제: 결제, 장바구니, 회원 관리 등 핵심 기능 누락
```

##### Case 2: Specifai 명세
```
프롬프트: [자동 생성된 1,000줄 상세 명세]
결과: 모든 기능이 포함된 완성도 높은 사이트
장점: 한 번에 정확한 구현, 재작업 최소화
```

## 2.4 문제 정의 요약

### 핵심 문제
1. **소통 문제**: 발주자와 개발자 간 언어 차이
2. **범위 문제**: 불명확한 프로젝트 범위 정의
3. **신뢰 문제**: 계약 근거 부재로 인한 분쟁
4. **효율 문제**: AI 시대에도 여전히 낮은 구현 효율

### Specifai가 해결하는 것

Specifai는 이런 페인포인트를 해결하려는 목적을 가지고 있습니다:

- ✅ **발주자를 위해**: 무엇을 만들고 싶은지 명확히 하기 위한 요구사항 구체화 지원
- ✅ **실무자를 위해**: 발주자의 명확한 요구사항 도출 지원
- ✅ **양측을 위해**: 요구사항명세서를 통한 사고 정렬 및 명확한 문서화
- ✅ AI 기반 대화로 **소통 격차 해소**
- ✅ 체계적 질문으로 **범위 명확화**
- ✅ 계약 문서 수준으로 **신뢰 구축**
- ✅ AI 프롬프트 생성으로 **효율 극대화**

**핵심 원칙**: 요구사항명세서가 구체화되고 명확할수록 양자간 만족할 수 있는 결과물을 얻을 수 있습니다.

### 시장 기회
```
문제의 크기 × 해결 가능성 × 타이밍 = 기회

크다 × 높다 × 완벽 = 💎 절호의 기회
```
