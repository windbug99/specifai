# Specifai MVP - UX 접근 방향성

**작성일:** 2025년 11월 12일  
**버전:** 1.0  
**목적:** 핵심 기능 검증을 위한 MVP UX 설계 방향 정립

---

## 1. 핵심 UX 원칙

### 1.1 기본 원칙

```
사용자 입력 → 객관식 선택 중심 → 빠른 결과 도출
```

**목표:**
- ✅ 최초 요구사항 입력 후 **객관식 선택만으로** 명세서 완성
- ✅ 질문 개수: **20개 이내** (기본), 최대 50개 (복잡한 프로젝트)
- ✅ 완료 시간: **10-30분** 이내
- ✅ 비개발자도 쉽게 답변 가능

### 1.2 동적 질문 흐름

프로젝트의 **규모, 주제, 구체화 수준**에 따라 질문 개수와 단계가 자동 조정됩니다.

| 프로젝트 유형 | 질문 개수 | 예상 시간 | 예시 |
|-------------|----------|----------|------|
| **단순** | 10-15개 | 10분 | 랜딩 페이지, 간단한 CRUD |
| **중간** | 15-20개 | 15-20분 | 소규모 쇼핑몰, 커뮤니티 |
| **복잡** | 20-30개 | 20-30분 | 다중 역할 플랫폼, 결제 시스템 |
| **매우 복잡** | 30-50개 | 30-45분 | 엔터프라이즈 솔루션 |

## 2. 사용자 컨텍스트 정보 (MVP 고정값)

### 2.1 컨텍스트 정보의 중요성

사용자 컨텍스트 정보를 수집하면 **질문의 범위, 깊이, 선택지**를 맞춤화할 수 있습니다:

| 컨텍스트 정보 | 영향도 | 활용 예시 |
|-------------|--------|----------|
| **발주자 역할** | 높음 | PM → 비즈니스 중심 질문<br>개발자 → 기술 중심 질문 |
| **실무자 경력** | 높음 | 5년차 → 전문 용어 사용 가능<br>주니어 → 쉬운 용어 사용 |
| **서비스 국가** | 중간 | 한국 → 카카오/네이버<br>미국 → Google/Apple |
| **예산 규모** | 중간 | 소규모 → 필수 기능 중심<br>대규모 → 부가 기능 포함 |

### 2.2 MVP 고정값 설정

현재 MVP에서는 다음 값을 **고정**하여 개발 복잡도를 낮춥니다:

```javascript
const MVP_CONTEXT = {
  requester: {
    role: "프로젝트 담당 PM",
    technical_knowledge: "중급", // 개발 용어 부분 이해
    company_size: "소규모"
  },
  developer: {
    role: "프리랜서 개발자",
    experience_years: 5,
    expertise: ["웹 개발", "풀스택"]
  },
  service: {
    country: "한국",
    language: "ko",
    timezone: "Asia/Seoul"
  }
}
```

### 2.3 컨텍스트 기반 최적화 예시

#### 예시 1: 소셜 로그인 선택지 (국가별)

```javascript
// 한국 서비스
options: [
  "카카오 로그인 (추천)", // 한국 1위
  "네이버 로그인",
  "구글 로그인",
  "애플 로그인"
]

// 미국 서비스 (향후)
options: [
  "Google 로그인 (추천)",
  "Apple 로그인",
  "Facebook 로그인",
  "Email only"
]
```

#### 예시 2: 질문 깊이 조정 (발주자 역할별)

```javascript
// PM 발주자 (비즈니스 중심)
질문: "사용자 인증은 어떻게 하실 건가요?"
선택지:
  - 이메일로만 가입
  - 소셜 로그인 (카카오, 네이버)
  - 전화번호 인증

// 개발자 발주자 (기술 중심) (향후)
질문: "인증 방식을 선택해주세요"
선택지:
  - JWT + Email/Password
  - OAuth 2.0 (카카오, 네이버)
  - SMS OTP + JWT
```

#### 예시 3: 용어 수준 조정 (실무자 경력별)

```javascript
// 5년차 프리랜서 (전문 용어 사용 가능)
설명: "RESTful API로 구현하며, 응답 시간은 200ms 이내입니다"

// 주니어 개발자 (향후)
설명: "서버와 통신하며, 빠른 응답 속도를 제공합니다"
```

## 3. UX 플로우 설계

### 3.1 전체 프로세스

```
0단계: 컨텍스트 설정 (MVP: 고정값)
   ↓
1단계: 초기 입력 (자유 서술형)
   ↓
2단계: AI 분석 & 프로젝트 분류
   ↓
3단계: 객관식 질문 (컨텍스트 기반 동적 생성)
   ↓
4단계: 실시간 명세서 생성
   ↓
5단계: 검토 & 수정
   ↓
6단계: 최종 문서 출력
```

### 3.2 단계별 상세 설계

#### **0단계: 컨텍스트 설정 (MVP: 자동 설정)**

**목적:** 질문 맞춤화를 위한 기본 정보 설정

**MVP 처리:**
```javascript
// MVP에서는 자동으로 설정 (사용자 입력 불필요)
const context = {
  requester: "프로젝트 담당 PM",
  developer: "프리랜서 5년차",
  country: "한국",
  language: "ko"
}

// 이 정보를 기반으로:
// - 질문 용어 수준 결정
// - 선택지 우선순위 결정 (예: 카카오 > 구글)
// - 설명 깊이 조정
```

**향후 확장 (Phase 2):**
사용자에게 4개 질문으로 컨텍스트 수집 (2-3분 소요)

---

**목적:** 사용자가 빠르게 선택만으로 요구사항 구체화

**질문 형식 3가지:**

- 한국 서비스 → 카카오/네이버 우선 표시

- 5년차 개발자 → 정확한 기술 정보 제공 ("+2일")

**질문 카테고리 구성 (예시: 이커머스)**

| 순서 | 카테고리 | 질문 수 | 예시 질문 |
|-----|---------|--------|----------|
| 1 | **기본 정보** | 2-3개 | 프로젝트명, 대상 사용자, 출시 목표일 |
| 2 | **회원 관리** | 2-3개 | 가입 방식, 로그인 방식, 권한 관리 |
| 3 | **핵심 기능** | 4-6개 | 상품 관리, 장바구니, 주문, 결제 |
| 4 | **부가 기능** | 2-4개 | 리뷰, 찜하기, 쿠폰, 알림 |
| 5 | **UI/UX** | 2-3개 | 디자인 스타일, 반응형, 접근성 |
| 6 | **기술/성능** | 2-3개 | 예상 사용자 수, 속도 요구사항 |
| 7 | **운영** | 1-2개 | 관리자 페이지, 통계 |

**질문 최적화 전략:**

1. **동적 질문 생성**
   - 이전 답변에 따라 다음 질문 결정
   - 불필요한 질문 자동 스킵
   - 예: 결제 기능 "아니오" → 결제 관련 질문 전체 스킵

2. **스마트 추천**
   - 비슷한 프로젝트 데이터 기반 추천
   - 예: "쇼핑몰 프로젝트의 85%가 '장바구니'를 선택했습니다"

3. **개발 영향도 표시**
   - 각 선택의 개발 시간, 비용 영향도 표시
   - 예: "+2일", "+50만원"

## 4. 동적 질문 생성 로직

### 4.1 질문 개수 결정 알고리즘

```
입력: 
  - 초기 사용자 설명
  - 컨텍스트 정보 (MVP: 고정값)
  
출력: 필요한 질문 개수 (10-50개)

1. 프로젝트 유형 분류
   - 쇼핑몰: 18-25개
   - 커뮤니티: 15-20개
   - SaaS: 20-30개
   - 랜딩 페이지: 10-12개

2. 복잡도 분석
   - 단순: 기본 질문 개수의 70%
   - 중간: 기본 질문 개수의 100%
   - 복잡: 기본 질문 개수의 150%

3. 사용자 구체화 수준
   - 매우 구체적 (상세한 초기 입력): -30%
   - 보통: 0%
   - 추상적: +20%

4. 컨텍스트 조정 (MVP: 고정)
   - PM 발주자: 비즈니스 질문 +20%
   - 5년차 개발자: 기술 깊이 질문 +10%
   - 한국 서비스: 국가별 특화 질문 포함

최종 질문 개수 = 기본 * 복잡도 * 구체화 * 컨텍스트
(최소 10개, 최대 50개)
```

### 5.2 컨텍스트 기반 질문 렌더링 로직

```javascript
function renderQuestion(question, context) {
  // 1. 질문 텍스트 선택
  const questionText = context.requester.role === "개발자"
    ? question.question_text.for_developer
    : question.question_text.for_pm || question.question_text.default;
  
  // 2. 옵션 설명 조정
  const options = question.options.map(opt => {
    const description = context.requester.role === "PM"
      ? opt.description.for_pm
      : opt.description.for_developer || opt.description.default;
    
    // 3. 국가별 소셜 플랫폼 필터링
    let socialPlatforms = [];
    if (opt.social_platforms_by_country) {
      socialPlatforms = opt.social_platforms_by_country[context.service.country] 
        || opt.social_platforms_by_country.default;
    }
    
    return {
      ...opt,
      description,
      socialPlatforms
    };
  });
  
  return {
    questionText,
    options
  };
}

// MVP 사용 예시
const MVP_CONTEXT = {
  requester: { role: "PM" },
  developer: { experience_years: 5 },
  service: { country: "korea" }
};

const rendered = renderQuestion(Q001, MVP_CONTEXT);
```

## 11. 결론

### 핵심 성공 요소

1. **컨텍스트 기반 맞춤화** ⭐ NEW
   - 발주자/실무자 역할에 따른 용어 조정
   - 서비스 국가에 따른 선택지 최적화
   - MVP: 한국 PM + 프리랜서 5년차 고정

2. **객관식 중심 UX**
   - 사용자 입력 최소화
   - 빠른 완료 (10-30분)
   - 비개발자 친화적

3. **동적 질문 흐름**
   - 프로젝트에 따라 10-50개
   - 불필요한 질문 스킵
   - 효율적 정보 수집

4. **실시간 피드백**
   - 진행률 표시
   - 즉시 명세서 생성
   - 비용/기간 예측

5. **AI-Native 결과물**
   - 계약서 수준 명세서
   - AI 도구 직접 사용 가능
   - 명확성 최대화

### MVP 핵심 제약사항

```
고정값 (하드코딩):
- 발주자: 프로젝트 담당 PM
- 실무자: 프리랜서 개발자 5년차
- 국가: 한국
- 언어: 한국어

이에 따른 최적화:
- 소셜 로그인: 카카오 > 네이버 > 구글 > 애플
- 결제: 토스페이먼츠 > 아임포트
- 용어: PM 수준 (비즈니스 중심, 기술 용어 최소화)
- 설명: 중급 개발자 이해 가능 수준

Phase 2 확장 시:
- 컨텍스트 입력 화면 추가
- 다국가 지원 (미국, 일본)
- 발주자 역할별 질문 분기
```
